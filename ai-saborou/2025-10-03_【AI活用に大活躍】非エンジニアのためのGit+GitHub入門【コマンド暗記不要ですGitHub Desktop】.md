あなたのローカルのメインブランチには
ない変更がリモートに含まれているわけ
です。取り込むにはまずローカルでメイン
ブランチに切り替えまして、
次に右横にあるプルボタンをクリックし
ます。
これでマージされた内容が取り込めました
。このようにブランチを駆使して
プロジェクトを管理するのがギットです。
1人で作業をしている時にはそこまで
ブランチは使わないかもしれませんが、
クラウドAIを活用し出すとAIに
プルリク作ってもらうようになるので是非
覚えてみてください。
これで基礎的な動作は一通りお伝えする
ことができました。最後に最初から必須と
いうほどではないですが、よく使う動作
たちをお伝えします。ギットを使っていく
中で困ったことがあった時に見返してみて
ください。1つ目はディスカードチェンジ
です。ディスカードチェンジは作業中の
ファイルの変更をなかったことにする機能
です。やっぱりこの変更いらないなと思っ
た時に使います。使い方は簡単で削除し
たいを右クリックしてリスカードを
クリックするだけです。行だけでなく
ファイル全体の変更を取り除くこともでき
ます。私は雑に書いたログを消す時などに
よく使ってます。2つ目はリバートです。
リバートは過去のコミットを打ち消す機能
です。このコミットやっぱいらんかったな
という時に使います。特定のコミットが
原因で障害が起きた時にウリバートみたい
な感じで使うことがちょくちょくあるので
思い出深い機能です。使う時はヒストリー
タブを開いてください。取り消したい
コミットを右クリックしてリバート
チェンジズインコミットを選択します。
するとそのコミットの変更を打ち消す
新しいコミットが作成されます。3つ目は
マージコンフリクト解消です。これは機能
というよりは複数の人が同じファイルの
同じ場所を編集した時に起こる問題です。
例えばあなたがブランチAで作業している
間に誰かがメインブランチの同じファイル
を編集したとします。そしてブランチAを
メインにマージしようとすると
コンフリクトが発生します。コンフリクト
が発生するとギitデスクトップの画面に
警告が表示されます。コンフリクト
ファイルとして問題のあるファイルが表示
されますね。エディターで該当のファイル
を開くとコンフリクトしている部分が特殊
な記号で囲まれています。ここではこの
記号を消してどちらの変更を採用するか、
あるいは組み合わせた別の業を作るかを
決めます。どうするかは状況により蹴りな
ので都度判断する必要があります。今回は
メインブランチの方を採用してみます。
エデターを使っていれば上の方にどちらか
を取り込むボタンがあるので、これを
クリックするだけで解決できます。
Gitッハブデスクに戻るとこんな感じの
グリーンな表記になっていると思います。
全てのコンフリクトを解決し
たらコンティニューマージを押してマージ
を完了させましょう。最後4つ目は
ドギットイグノアです。Gitノアはその
中に書かれたファイルやフォルダーの名前
のものがギッに追加できなくなるという
特殊なファイルです。Gitッignを
使う主なケースを紹介するために絶対に
注意して欲しいことを1つ申し上げます。
これはAPIキーなどの機密情報を絶対に
Gitubにアップロードしないで
くださいということです。GitHubは
開発に使うユースケースがメインな特性上
、APIキーなどを扱うプロジェクトが
多いです。レポジトリをプライベートにし
ていれば即座の被害は特にはないと思い
ますが、パブリックレポジトリだとその
APIキーが悪意のある人に勝手に使われ
てしまって高額な請求が来るなんてことに
なりかねません。これを防ぐためにはまず
このような機密情報はド部という環境変数
をまとめたファイルに書き込み、そして
レポジトリの中にgidッignファイル
を用意します。Gイグノアファイルの中に
ド部と書き込むとド部ファイルが
レポジトリの中に含まれることを防ぐこと
ができます。コーディングをしてこういっ
た環境変数を使う際には絶対にこの設定を
しておいてください。以上、よく使う
ティップスをお伝えしました。これらは
最初から全て覚える必要はありませんが、
いざという時に思い出せるようにしておく
と便利です。
